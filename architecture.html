Fig. 3 — RAG Knowledge Engine Architecture
Internal architecture of the Retrieval-Augmented Generation engine (MetadataRAGEngine). Shows the offline indexing phase (document loading, chunking, embedding) and the online retrieval phase (query encoding, FAISS search, metadata filtering).

Knowledge Document Store

Document Loader
os.walk() · .md files · extract doc_type, role, domain, skill metadata from path structure
↓
Text Chunker
Split by word count (chunk_size = 300) · attach metadata copy to each chunk · store in text_chunks[ ] + metadata[ ]
↓
Sentence Encoder
all-MiniLM-L6-v2 (SentenceTransformers)
Output: 384-dimensional dense vectors
↓
🗄️ FAISS Index
IndexFlatL2 · d=384
In-memory · built at startup
Singleton pattern
ONLINE — Query Retrieval Phase
Query Input
e.g. "backend_engineer web_development ATS requirements" or "docker kubernetes microservices"
↓
Query Encoder
Same all-MiniLM-L6-v2 model
Output: 384-dim query vector
↓
FAISS Search
index.search(query_vec, top_k × 2)
Returns distances + indices
↓
Metadata Filter
Filter by:
• role (e.g. "backend_engineer")
• domain (e.g. "web_development")
• doc_type (e.g. ["roles","ats","domains"])
↓
📤 Retrieved Context Chunks
Top-K grounded text passages
Injected into LLM prompt as context
Figure 3. Architecture of the Retrieval-Augmented Generation (RAG) knowledge engine. The system uses an offline indexing phase to embed domain-specific markdown documents into a FAISS vector index, and an online retrieval phase to fetch semantically relevant, metadata-filtered context chunks for grounding LLM responses.
